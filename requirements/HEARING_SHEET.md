# ヒアリングシート（HEARING_SHEET）

このシートは **要件定義の入力フォーム** です。未記入がある場合、AIエージェントはユーザーへ質問して埋めます（推測で埋めない）。

- 本シートは **機能要件**（実装に直接関わるもの）と **非機能要件**（運用・パフォーマンス等。リリース前後に決めてもよいもの）に分けて記載します。
- しつこい再質問を避けるため、ユーザーが「未定/TBD」と回答したらそのままTBDと記載し、決定期限がわかれば併記します。
- 機能要件はおおむね **0〜5章**、非機能要件は **6〜10章** を想定しています。
- TBDの表記ルール: 「まだ未質問」は `TBD（未質問）`、ユーザーに聞いて保留になったら `TBD（確認済/理由/期限）` とメモする（再質問防止のため）

- 目的: コーポレートサイト構築に必要な要件を漏れなく確定する
- 出力: ヒアリング結果は仕様ディレクトリに保存する（`HEARING_RULES.md` §1.3）
  - spec-kit + NWG: `specs/<NNN>-<name>/requirements.md`
  - WSK（spec-kit なし）: `docs/projects/<project_slug>/REQUIREMENTS_<project_slug>.md`
  - テンプレ `REQUIREMENTS_TEMPLATE.md` を複製して作成

（作成日: 2026-01-27 JST）

---

## 記入ルール

- 分からない項目は `TBD` と書く（空欄禁止）
- **AIエージェントは、要件定義フェーズに限り質問を「一問一答」形式で行う（一度に大量の質問を列挙しない）。合意済み事項は以後繰り返し質問しない。**
- 重要度は `Must / Should / Could / Won't` のいずれか
- “国際標準 / Google推奨 / 定石”がある場合はそれを優先する
- APIキー・トークン等の秘密はシートに直接記載しない。取得状況と保管場所のみ記入し、実キーはローカルの `.env.tmp`（Git管理外）に記録する。未取得はTBDのまま残し、実装開始前に再確認する。

## 2.1 外部サービス連携

**※ §2 で Google Maps「表示する」と回答した場合、以下を確認する。**

| # | 質問 | 回答 | 重要度 |
|---|---|---|---|
| 1 | Google Maps の埋め込み方法（iframe / Maps JavaScript API） | TBD | Must |
| 2 | Maps API キーの取得状況（取得済 → `.env.tmp` に記録 / 未取得） | TBD | Must |
| 3 | 表示する住所・所在地（日本語） | TBD | Must |
| 4 | 地図の表示スタイル（デフォルト / カスタムスタイル） | TBD | Should |

**埋め込み方式の判断基準:**
- **iframe 方式**: API キー不要。Google Maps の「共有 > 地図を埋め込む」で取得した iframe コードを使用。カスタマイズ不可だがコスト 0。CSP で `frame-src https://www.google.com` の許可が必要。
- **Maps JavaScript API 方式**: API キー必要。スタイル変更・マーカーカスタマイズ可。月間 28,500 回のマップ読み込みまで無料（$200 相当のクレジット）。

**API キー取得手順（Maps JavaScript API 方式の場合）:**
1. [Google Cloud Console](https://console.cloud.google.com/) でプロジェクトを作成（または既存を選択）
2. 「API とサービス」→「Maps JavaScript API」を有効化
3. 「認証情報」でAPIキーを発行
4. キーの制限を設定: HTTP リファラー制限で本番ドメインのみ許可（不正利用防止）
5. 発行したキーを `.env.tmp` に記録（Git 管理外）

**実装時の管理方針:**
- API キーは環境変数 `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` で管理する（`DEV_RULES §4.1` 参照）
- 住所・所在地は定数ファイルで管理する（`DEV_RULES §2.8` 準拠）

| （作成する場合）作成方法：A) ユーザー提供（Figma等URL）/ B) 外部ツールで生成して取り込み（Google Stitch等）/ C) AIエージェントが静的HTMLモックで作成 | TBD | Should |

**ワイヤーフレーム忠実度の判断基準（`DESIGN_RULES §8.0`）:**
- **簡易版**（Lo-fi）: グレーボックスとテキストラベルで構造・導線を確認する。デザイナーが後続でビジュアルデザインを仕上げる、または方向性を早期確認したい場合に適する。**合意がない場合のデフォルト**。
- **詳細版**（Hi-fi）: カラー・フォント・実画像を含む、ほぼ完成版の画面。AIが最終デザインまで担当する場合や、デザイナーがいないプロジェクトに適する。

**ワイヤーフレーム作成方法の判断基準:**
- **A) ユーザー提供**: ユーザーがFigma等で作成済みのワイヤーフレームがあればそれを基に実装する。
- **B) 外部ツールで生成**: Google Stitch等のAIデザインツールで生成したワイヤーフレームを取り込む場合。出力形式に応じて、画像ならデザインリファレンスとして参照、HTML/コード出力なら `docs/wireframes/` に配置してAIエージェントが実装に取り込む。
- **C) AIエージェントが作成**: ユーザー提供も外部ツール出力もない場合、AIエージェントがリポジトリ内に静的HTMLモックとして作成・配置する。ユーザーにFigma等での新規作成を要求しない。

※いずれの方法でも、ワイヤーフレーム/UIモック作成時は `DESIGN_RULES.md` のUI/UX設計原則（ISO 9241-110 / Nielsen ヒューリスティクス / レイアウト・レスポンシブ・導線設計原則）に準拠すること。
